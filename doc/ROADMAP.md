# Bakubaku 项目计划与路线图

## 项目信息

- **项目名称：** Bakubaku（吞吞果实）
- **技术栈：** Rust（核心引擎）+ Tauri 2 + React + TypeScript（GUI）
- **开发模式：** 个人项目，两周一个迭代

---

## 阶段划分

### Phase 0 — 技术验证（2 周）

**目标：** 验证两个核心假设 — 扫描速度和 LLM 分析准确率。

**交付物：** 一个 Rust CLI 原型，能扫描目录并输出 LLM 分析结果。

| 任务 | 状态 | 说明 |
|------|------|------|
| 项目初始化（Rust workspace） | 待开始 | Cargo workspace 结构搭建 |
| 文件扫描 PoC | 待开始 | 用 jwalk/NTFS MFT 读取实现快速遍历，输出文件元数据 |
| LLM 分析 PoC | 待开始 | 批量发送文件路径给 Claude API，验证分析质量 |
| 准确率评估 | 待开始 | 扫描自己的磁盘，人工对比 LLM 判断结果 |

**Phase 0 成功标准：**
- [ ] 100 万文件扫描 < 10 秒（SSD）
- [ ] LLM 对文件重要性判断准确率 > 85%
- [ ] 成本评估：全盘分析 API 费用 < $0.50

## 技术架构

```
bakubaku/
├── crates/
│   ├── bakubaku-core/       # 核心引擎库
│   │   ├── scanner/         # 扫描引擎
│   │   ├── rules/           # 规则引擎
│   │   ├── analyzer/        # LLM 分析模块
│   │   ├── classifier/      # 三层分类整合
│   │   ├── memory/          # 记忆系统
│   │   ├── cleaner/         # 清理执行
│   │   └── logger/          # 操作日志
│   └── bakubaku-cli/        # CLI 界面
├── gui/                     # Tauri + React GUI
│   ├── src/                 # React 前端
│   └── src-tauri/           # Tauri Rust 后端
├── rules/                   # 规则配置文件
├── doc/                     # 项目文档
└── tests/                   # 集成测试 + 沙箱测试数据
```

**核心依赖（Rust）：**
- `jwalk` — 并行文件遍历
- `ntfs` — NTFS MFT 直读
- `rusqlite` — SQLite 记忆系统
- `reqwest` — LLM API 调用
- `serde` — 序列化
- `tokio` — 异步运行时
- `clap` — CLI 参数解析

**GUI 依赖：**
- `tauri 2.x` — 桌面应用框架
- `react 18` — 前端框架
- `typescript` — 类型安全
- `recharts` 或 `d3` — 数据可视化

## 质量保障

### 测试策略

1. **沙箱测试环境** — 创建标准化的测试目录结构，包含各种文件类型，每次测试在沙箱中验证清理结果
2. **硬编码保护名单** — 系统目录、用户文档目录等绝不触碰，不依赖 LLM 判断
3. **隔离区兜底** — 所有删除先进隔离区，保留 7 天
4. **LLM 置信度阈值** — 置信度 < 90% 一律标记为"需人工确认"
5. **回归测试** — 每次迭代确保已有功能不退化

### 安全红线

- 永远不直接永久删除文件（除非用户主动清空隔离区）
- 永远不删除系统关键文件（硬编码白名单）
- 永远不上传文件内容到云端
- 永远不后台运行或自启动
